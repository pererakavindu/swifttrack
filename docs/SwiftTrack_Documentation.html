1. Introduction & Problem Statement

SwiftLogistics is a last-mile delivery company operating at scale across Sri Lanka. The company relies on three independent, legacy systems that have been developed separately over time and use heterogeneous communication protocols:

Table 1: Existing Systems Overview
SystemProtocolData FormatPurpose
Client Management System (CMS)SOAP over HTTPXMLOrder intake, customer records, billing
Route Optimisation System (ROS)REST over HTTPJSONCloud-based delivery route planning
Warehouse Management System (WMS)Proprietary TCP/IPPipe-delimitedInventory tracking, package operations

These systems operate in isolation with no automated data flow between them. Order information must be manually transferred between departments, route planning cannot access real-time warehouse availability, and clients have no visibility into delivery progress. During peak periods such as Black Friday and Avurudu season, this manual coordination creates significant bottlenecks, delayed deliveries, and customer dissatisfaction.

The goal of the SwiftTrack platform is to design and implement a middleware architecture that integrates these three heterogeneous systems into a unified, real-time logistics platform featuring a web-based Client Portal and a mobile-style Driver Application.

2. Architectural Challenges

Based on a thorough analysis of the business scenario, we identified six critical architectural challenges that our middleware must address:

Challenge 1 — Heterogeneous System Integration: The three backend systems use entirely different communication paradigms (SOAP/XML, REST/JSON, TCP/pipe-delimited). The middleware must abstract these differences and present a unified API surface.
Challenge 2 — Real-Time Tracking: Clients need live visibility into delivery status. Drivers need instant manifest updates. This requires a persistent, bidirectional communication channel (WebSocket) backed by an event-driven architecture.
Challenge 3 — High-Volume Asynchronous Processing: During peak sales events, thousands of orders may be submitted simultaneously. The architecture must decouple order intake from processing to prevent system overload using message queues.
Challenge 4 — Distributed Transaction Management: A single order placement spans three systems (CMS → WMS → ROS). If any step fails, all previously completed steps must be rolled back. This requires a saga pattern with compensating transactions.
Challenge 5 — Scalability & Resilience: Individual services must be independently deployable and scalable. Failure in one service must not cascade to others.
Challenge 6 — Security: The platform must authenticate users, enforce role-based access control (client/driver/admin), validate all inputs, and prevent injection attacks — especially on the proprietary TCP protocol.

3. Alternative Architectures Considered

Before arriving at our final architecture, we evaluated three distinct approaches:

Option A: Monolithic Enterprise Service Bus (ESB)

A traditional ESB approach would use a single, centralized middleware bus (e.g., Apache ServiceMix or WSO2 ESB) to mediate all communication. While this offers centralized configuration and transformation, it introduces a single point of failure, is difficult to scale horizontally, and creates tight coupling between the bus and all services. Given the requirement for high-volume async processing and real-time events, this architecture would struggle at scale.

❌ Rejected — Single point of failure, poor scalability for high-volume events.

Option B: Pure Microservices with Choreography

In a choreography-based approach, each service reacts to events independently — there is no central coordinator. Services publish and subscribe to events via a message broker. While this is highly decoupled, the distributed transaction spanning CMS → WMS → ROS becomes very difficult to manage, debug, and reason about. Compensating transactions would require complex event chains with no single point of visibility.

⚠️ Rejected — Saga rollbacks across three systems are error-prone without a central orchestrator.

Option C: API Gateway + Orchestrated Microservices + Event-Driven (Selected)

Our chosen architecture combines the best aspects of centralized orchestration and event-driven decoupling:

  • An API Gateway provides a single entry point for all client requests, handling authentication, routing, and WebSocket management.
  • A dedicated Orchestrator service implements the saga pattern, managing the multi-step order transaction with explicit compensation logic.
  • Adapter services translate between the gateway's REST/JSON interface and each backend's native protocol.
  • RabbitMQ handles asynchronous order queuing for high-volume scenarios.
  • Redis provides fast state storage and Pub/Sub for real-time WebSocket event distribution.

✅ Selected — Best balance of centralized control, async capability, and real-time events.

5. Architectural & Integration Patterns

5.1 Saga Pattern — Distributed Transaction Management

The order placement process spans three independent systems. Since distributed ACID transactions are not feasible across SOAP, TCP, and REST services, we implement the Saga Pattern (Orchestration variant) to maintain eventual consistency with compensating transactions.

Figure 4: Saga Pattern — Order Placement with Compensating Transactions

5.2 Adapter Pattern — Protocol Translation

The Adapter Pattern is used to decouple the middleware's internal REST/JSON interface from the heterogeneous protocols of the legacy backend systems:

Table 2: Protocol Adapters
AdapterInternal InterfaceExternal ProtocolTranslation Logic
CMS AdapterREST/JSON (FastAPI)SOAP/XML (Spyne)Zeep SOAP client marshals JSON→XML request, parses XML→JSON response
WMS AdapterREST/JSON (FastAPI)Proprietary TCP/IPasyncio TCP client builds pipe-delimited commands, parses pipe-delimited responses
ROS (no adapter)Direct REST/JSON — no translation required

5.3 Publish-Subscribe Pattern

Two Pub/Sub mechanisms are employed:

  • RabbitMQ Topic Exchange (swifttrack.events): Used for durable, persistent event distribution between microservices. The order.submissions queue enables asynchronous order processing — orders are published by the gateway and consumed by the orchestrator in the background.
  • Redis Pub/Sub (swifttrack:tracking channel): Used for lightweight, real-time event broadcasting. The orchestrator publishes status updates, and the gateway's Redis subscriber pushes these to connected WebSocket clients with intelligent routing based on user role (client, driver, admin).

8. Security Architecture

Security is implemented at every layer of the architecture:

Table 6: Security Controls
ControlImplementationLayer
AuthenticationJWT tokens (HS256 algorithm, 60-minute expiry). Tokens issued on login and validated on every API request via Depends(get_current_user).Gateway
Role-Based Access ControlThree roles: client, driver, admin. Each endpoint enforces role checks. Clients can only view their own orders; drivers can only update deliveries.Gateway
Input ValidationPydantic models validate all request bodies with type checking, required fields, and constraints.All services
TCP Injection PreventionThe sanitize_field() function strips pipe (|) and newline characters from data before constructing WMS TCP commands, preventing protocol injection attacks.Mock WMS
Error SanitizationInternal service URLs and stack traces are never exposed in API error responses. Generic error messages are returned to clients.Gateway, Adapters
Message PersistenceRabbitMQ messages use delivery_mode=PERSISTENT, ensuring no message loss on broker restart.Orchestrator
Network IsolationAll services run on a private Docker bridge network (swiftnet). Only the API Gateway port (8000) is exposed to the host.Infrastructure
WebSocket SecurityWebSocket connections require a valid JWT token as a query parameter. Expired or invalid tokens result in connection rejection.Gateway
CORSCross-Origin Resource Sharing is configured on the API Gateway to control which origins can access the API.Gateway

  1. Synchronous Order: Login as megamart → Submit order → Observe PENDING → ACCEPTED → IN_WAREHOUSE → PROCESSING status transitions in real-time without page refresh.
  2. Asynchronous Order: Submit via "Submit Async" → Observe QUEUED status returned immediately → Watch saga complete in background → Real-time updates via WebSocket.
  3. Driver Delivery: Login as driver1 on Driver App → See manifest with assigned orders → Mark as DELIVERED → Verify Client Portal updates instantly.
  4. Saga Compensation: If WMS or ROS fails during order processing, observe compensating transactions roll back previously completed steps.
  5. WebSocket Real-Time: Open Client Portal and Driver App side-by-side → Submit order from client → Watch both portals update simultaneously.

10. Conclusion

The SwiftTrack middleware platform successfully addresses all six architectural challenges identified in the business scenario:

Table 7: Challenge Resolution Summary
ChallengeSolutionTechnology
Heterogeneous IntegrationAdapter Pattern — protocol translationZeep (SOAP), asyncio (TCP), httpx (REST)
Real-Time TrackingEvent-driven WebSocket hubRedis Pub/Sub → WebSocket
High-Volume Async ProcessingMessage queue decouplingRabbitMQ persistent queues
Distributed TransactionsSaga Pattern with compensationOrchestrator service
Scalability & ResilienceContainerized microservicesDocker Compose, health checks
SecurityMulti-layered auth & validationJWT, RBAC, Pydantic, input sanitization

The architecture leverages exclusively open-source technologies and follows established enterprise integration patterns (Adapter, API Gateway, Saga, Publish-Subscribe) to create a modular, scalable, and maintainable middleware solution. The containerized deployment via Docker Compose enables reproducible builds and simplified operations, while the event-driven architecture ensures real-time visibility across all stakeholders.
